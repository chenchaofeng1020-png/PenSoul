## 问题原因
- 本地开发（非 Supabase）模式下，前端在接受邀请时从 `localStorage` 读取 `username/email`，为空时会用占位值（例如“用户”或 `user@example.com`），导致后端将该占位值写入 `product_members`。
- 产品列表接口 `GET /api/products` 使用 `?user=` 参数仅按成员的 `username/email` 过滤；占位值与真实登录用户不匹配，成员看不到任何产品。
- 证据：
  - 接受邀请写入占位：`server/index.js:1073–1083`
  - 产品列表按 `?user=` 过滤：`server/index.js:67–79`
  - 前端邀请页取值来源为 `localStorage`：`src/components/InvitationPage.jsx:38–75`
  - 前端产品列表在非 Supabase 模式仅传 `?user=`：`src/services/api.js:338–346`

## 修复方案
1. 前端：邀请页使用真实登录信息
- 在 `InvitationPage.jsx` 中改为通过 `getUser()` 读取 `user.email` 与 `user.user_metadata.username`，禁止使用 `localStorage` 占位；为空则提示并阻止提交。
- 位置：`src/components/InvitationPage.jsx:38–75`。

2. 后端：放宽匹配并支持用户 ID
- `GET /api/products` 增加对 `full_name` 的匹配，且支持 `uid` 参数（用户 ID），同时在过滤时命中 `m.id === uid`。
- 位置：`server/index.js:67–79`。

3. 前端：同时传递用户 ID
- 在 `getProducts()` 中，非 Supabase 模式追加 `&uid=${user.id}`，保持兼容原有 `?user=`。
- 位置：`src/services/api.js:338–346`。

4. 数据一致性修复（已有受邀占位成员）
- 接受邀请时，若该成员已存在且占位（如 `email` 为 `user@example.com`），根据当前真实用户信息更新该成员的 `username/email`。
- 位置：`server/index.js:1101–1114`（在已有升级逻辑旁新增字段纠正）。

## 验证步骤
- 创建邀请 → 无痕窗口注册/登录 → 接受邀请。
- 刷新产品列表：左侧“请选择产品”应展示受邀产品；日志显示匹配命中。
- 交叉验证：
  - 非 Supabase：`getProducts()` 请求含 `user` 与 `uid`；后端返回含受邀产品。
  - Supabase：原逻辑不变（成员通过 `product_members.user_id` 合并），保持兼容。

## 风险与回滚
- 改动均为兼容增强；若出现异常，去除 `uid` 支持并仅保留前端真实信息写入即可回退。
