## 问题定位
- 前端“团队成员”页面只在成员列表包含当前用户且角色为 owner/admin 时显示“邀请成员”按钮。
- 新创建产品的接口返回 `data: []`，导致角色判定为 null → 按钮隐藏。
- 造成空列表的两个路径：
  1) Serverless: `/api/products/[productId]/members/index.js` 仅读取 `product_members`，成员表为空时返回空。
  2) 本地 Node: `server/index.cjs` 读取本地 `db.json`，当使用 Supabase 创建产品或成员未写入本地时也可能为空。

## 修复方案
1. 补充成员接口兜底（Serverless）
- 在 `api/products/[productId]/members/index.js`：
  - Supabase 分支：若 `product_members` 查询为空，读取 `products.owner_id`，用 `auth.admin.getUserById(owner_id)` 补齐 `email/username/full_name`，返回一条 `role: 'owner'` 的成员。
  - 本地 JSON 分支：若 `db.product_members[pid]` 为空，读取 `db.products` 的 `owner_id`，合成一条 `owner` 成员返回。

2. 保持本地 Node 接口一致
- 已有 `server/index.cjs` 的 GET `/api/products/:id/members`，再确认添加同样的“owner 兜底”逻辑与字段映射（email/username/full_name）。

3. 确认产品创建后写入成员（可选增强）
- 为避免成员表长期为空：
  - 将前端的产品创建改为调用 Serverless `POST /api/products`（服务端用 service_role 写入 `product_members`），或
  - 新增 `POST /api/products/:id/members/owner-seed` 由服务端补种 owner（service_role），前端在创建后调用。

4. 前端无需改动 UI 判定逻辑
- `TeamMemberPage.jsx` 已兼容返回 `data: []` 或 `data: { members: [] }`；只要接口返回包含 `owner` 的成员项，按钮即可显示。

## 验证步骤
- 本地：打开 `http://localhost:5173/api/products/<pid>/members`，看到至少一条 `owner`。
- 页面：进入“团队成员”，显示“邀请成员”按钮；列表展示“产品创建人”。
- 线上：同样验证 `/api/products/<pid>/members` 与页面行为。

## 风险与兼容
- 使用 service_role 查询用户资料，仅用于服务端；不会影响前端安全。
- 若 Supabase 用户资料缺少 `username/full_name`，用邮箱或空字符串兜底，不影响按钮显示（依邮箱匹配）。

## 执行计划
1) 更新 `api/products/[productId]/members/index.js` 增加 owner 兜底逻辑
2) 对齐并复查 `server/index.cjs` 的成员兜底实现与字段映射
3) （可选）将前端产品创建改为调用 Serverless `POST /api/products`，保证成员写入由服务端完成
4) 自测本地与线上接口与页面显示
