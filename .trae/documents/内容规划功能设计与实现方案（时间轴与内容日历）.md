## 背景与目标
- 背景：面向个人开发者/小团队，产品上线后缺乏系统化推广规划，导致优质产品难以获得增长。
- 目标：在现有产品中增加“内容规划”能力，支持用户创建推广内容计划，在指定平台于指定时间发布；提供“时间轴视图”和“内容日历视图”以便查看、管理与协作；保持现有 UI 风格与交互一致。

## 使用者与场景
- 使用者：
  - 创始人/独立开发者：个人制定与跟踪推广计划。
  - 小团队市场/运营：多人协作、分配任务与审核内容。
- 核心场景：
  - 计划编排：选择平台、时间、内容类型与素材，形成计划条目。
  - 执行跟踪：时间轴/日历查看进度，状态切换（草稿/待发布/已发布/延期）。
  - 协作：负责人、@成员、评论与附件。
  - 提醒：计划前提醒（邮件/站内），逾期告警，重复计划。

## 用户故事
- 作为用户，我可以新建一个“内容计划”，为某产品选择目标平台、发布时间、标题与正文，并上传配图。
- 作为用户，我可以在“时间轴视图”按时间顺序查看计划，快速筛选平台与状态。
- 作为用户，我可以在“内容日历视图”按周/月查看每天的内容安排，并拖拽调整日期/时间。
- 作为用户，我可以为计划指定负责人，并在计划临近时收到提醒。
- 作为用户，我可以为重复活动（如“每周一在 X 发布更新日志”）设置重复规则。
- 作为用户，我可以记录实际发布链接与数据（阅读/点赞）以便复盘。

## 信息架构
- 新导航项：内容规划（与现有“竞品管理/产品路线图/团队成员/系统截图”并列；保持一致风格与尺寸）。
- 页面结构：
  - 顶部筛选：产品切换、平台筛选、状态筛选、时间范围切换（周/月）。
  - 视图切换：时间轴 / 日历（按钮切换，样式沿用现有 UI 组件逻辑）。
  - 右上角主按钮：新建计划。
  - 详情抽屉：计划条目点击后在右侧抽屉显示详情与操作（编辑、评论、附件、状态变更）。

## 数据模型（Supabase）
- `content_plans`（计划集合，可按产品聚合）
  - `id uuid pk`
  - `owner_id uuid`（RLS：= auth.uid）
  - `product_id uuid`（关联产品）
  - `name text`（如“9月推广计划”）
  - `start_date date`、`end_date date`（可选）
  - `created_at timestamptz`
- `content_items`（具体计划条目）
  - `id uuid pk`
  - `plan_id uuid`（关联 `content_plans`）
  - `product_id uuid`
  - `platform text`（枚举：`x`、`zhihu`、`bilibili`、`wechat_mp`、`juejin`、`toutiao`…）
  - `title text`、`body text`
  - `assets jsonb`（图片/视频存储路径数组）
  - `schedule_at timestamptz`（计划发布时间）
  - `repeat_rule jsonb`（如 `{freq:'weekly', byDay:['MO']}`）
  - `status text`（`draft`/`scheduled`/`posted`/`delayed`）
  - `assignee_id uuid`（负责人）
  - `actual_url text`（实际发布链接）
  - `metrics jsonb`（阅读/点赞等）
  - `created_at timestamptz`
- `content_item_comments`
  - `id uuid pk`、`item_id uuid`、`author_id uuid`、`body text`、`created_at`
- `content_item_reminders`
  - `id uuid pk`、`item_id uuid`、`remind_at timestamptz`、`channel text`（`email`/`in_app`）
- `content_item_assets`（可选：若不使用 `jsonb`）
  - `id uuid pk`、`item_id uuid`、`storage_path text`、`created_at`
- Storage：桶 `content-assets`（私有，签名 URL 展示）

## RLS 策略（与现有产品一致）
- 所有表 `enable row level security`。
- 选择策略：
  - `content_plans`：`owner_id = auth.uid()` 读写。
  - `content_items`：关联产品归属校验（`product_id in (select id from products where owner_id = auth.uid())`）。
  - `comments/reminders/assets`：通过 `item_id -> content_items -> product_id` 归属校验。

## 交互与流程
- 新建计划：点击“新建计划”→ 弹窗（选择产品/平台/时间/标题/正文/素材）→ 保存为 `content_items`（可选：选择或创建 `content_plans`）。
- 时间轴视图：按 `schedule_at` 升序分组显示；支持平台与状态筛选；条目 hover 显示操作（编辑/状态/删除）。
- 日历视图：周/月切换；支持拖拽改日期/时间（更新 `schedule_at`）；显示条目卡片（平台图标、标题）。
- 详情抽屉：
  - 基本信息：平台/时间/状态/负责人/素材缩略图/实际链接。
  - 行为：编辑、添加评论、上传附件、设置提醒、变更状态。
  - 复盘：填充实际链接与数据（手动录入或爬取，先做手动）。
- 重复计划：
  - 在新建/编辑中选择重复规则；系统生成下一周期条目或在渲染层展开（首期先以复制生成多条）。
- 提醒：
  - `content_item_reminders` 定时器：Vercel Cron（生产）或前端提示（本地）；到点触发邮件或站内通知。

## 视图与组件（保持现有风格）
- 顶部工具栏：沿用现有按钮与筛选组件风格（如圆角、渐变、阴影）。
- 时间轴组件：卡片列表，左侧时间节点，右侧内容卡片（平台图标 + 标题 + 状态标识）。
- 日历组件：周/月格子；条目卡片含平台徽标色；支持拖拽（后续迭代）。
- 详情抽屉：与现有 `FeatureAnalysisDetailDrawer` 风格一致，右侧抽出。

## API 设计（Vercel Functions）
- `GET /api/content/plans?productId=...`：列出计划集合（可选）。
- `POST /api/content/plans`：创建计划集合。
- `GET /api/content/items?productId=...&from=...&to=...&platform=...&status=...`：列表。
- `POST /api/content/items`：创建条目。
- `PUT /api/content/items/:id`：更新条目（时间、平台、内容、状态、负责人）。
- `DELETE /api/content/items/:id`：删除。
- `POST /api/content/items/:id/reminders`：新增提醒。
- `POST /api/content/assets/upload`：上传素材（返回 `signedUrl`）。

## 技术实现
- 前端（Vite + React）：
  - 新页面路由：`/planning`（内容规划），不影响现有页面结构。
  - 视图切换与筛选：使用现有按钮与选择器组件模式；日历可先用自研简历格子，后续再引入库。
  - 数据层：沿用 `src/services/api.js` 结构，新增 `getContentItems/createContentItem/updateContentItem/...`。
- 后端（Vercel Functions + Supabase）：
  - 所有接口使用服务密钥校验 `Authorization: Bearer <token>` 获取 `auth.uid()`；操作 Supabase 表。
  - 提醒：生产使用 Vercel Cron 调用 `api/cron/reminders` 扫描 `remind_at <= now()` 并发送邮件（后续）。
- 存储：`content-assets` 桶，上传返回签名 URL；前端展示缩略图。

## 权限与安全
- 所有表开启 RLS；仅资源归属用户可访问。
- 上传文件类型与大小限制；拒绝可执行与危险 MIME。
- 令牌过期处理与重登。

## 指标与度量
- 计划量、执行率（按期发布率）、逾期率、平台分布。
- 内容效果（手动录入）：点击/阅读/点赞。
- 复盘维度：标题长度、素材数量与发布时间与效果的相关性（后续）。

## 迭代拆分
- 里程碑 1：基础模型 + 新建条目 + 时间轴视图 + 日历视图（只读） + 素材上传。
- 里程碑 2：日历拖拽改期 + 提醒（站内） + 重复规则（复制生成）。
- 里程碑 3：邮件提醒（Vercel Cron）+ 指标录入 + 简单分析报表。

## 兼容与扩展
- 平台枚举可配置；未来支持平台 API 直连自动发布（需要 OAuth 与各平台 API，后续阶段）。
- 多人协作：使用现有团队成员与邀请机制，关联 `assignee_id` 与评论。

## 风险与规避
- 日历交互复杂度：先以基础格子实现，再迭代拖拽。
- 邮件服务：开发阶段不启用邮箱确认与邮件提醒，生产阶段再配置 SMTP。
- API 直连平台：先不做，避免复杂授权与风控。

## 验收与交付
- 新导航“内容规划”可进入；新建条目成功；时间轴与日历可查看；素材上传可用；RLS 正常限制；在无邮箱确认下可登录并使用。
