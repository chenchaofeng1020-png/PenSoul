## 总体目标

* 每个 Agent 具备“人类式”自主：能判断何时调用工具、何时请求其他 Agent 协作；在结果不确定时主动向用户确认。

* 形成统一的工具生态与协作协议，保证可扩展、可观测、可控。

## 架构概览

* **AgentRuntime 扩展**：在 [AgentRuntime.js](file:///Users/code/PenSoul/src/services/agent/AgentRuntime.js) 增加“工具注册表 + 决策器”。

  * 工具注册表：声明工具能力、输入输出模式、时效性、成本、可靠性。

  * 决策器（Policies）：基于用户意图、上下文、置信度、时效要求，决定“调用工具/请求协作/直接回答/询问确认”。

* **黑板与事件总线**：前端会话充当黑板（消息状态），后端记录协作轨迹；各 Agent 通过“系统消息”进行协作（例如诸葛→卓伟）。

* **协作模式**：

  * 去中心化优先：发起方 Agent 根据策略自决；必要时由“轻量 Planner”仲裁冲突（如预算超限）。

## 决策策略（Policies）

* **意图分类**：判断是否为“实时/事实检索、创作策划、数据分析、代码/结构化输出”等。

* **置信度阈值**：

  * 高置信度：直接回答或给出选题建议。

  * 中置信度且涉及时效：调用“联网搜索/卓伟”。

  * 低置信度或风险：发起**用户确认**（一句话澄清，列出将采取的动作与影响）。

* **时效性规则**：最近两年数据优先；无来源/来源过旧则再次检索或提示“可能过时”。

* **成本与预算**：每轮会话设置最大工具调用次数/时长；超限时提示用户是否继续。

## 工具生态

* **工具声明**（Tool Manifest）：

  * name、能力描述、输入 schema、输出 schema、ETA、侧效（网络/费用）、可靠性分。

* **统一调用协议**：

  * Agent 输出 JSON 指令（tool\_call），前端/后端执行并将结果回灌为系统消息。

  * 对不确定场景，Agent 输出 confirm\_request（包含理由、预计收益/成本），等待用户“是/否”。

* **内置工具**：

  * research（联网搜索，返回 sources + report 仅内部使用）。

  * browse（轻量网页预览/提取要点）。

  * summarize（结构化摘要）。

## 多 Agent 协作

* **诸葛（策划）**：

  * 先判定是否需要 research/browse；只在聊天栏展示来源列表；然后简要总结 + 3 个选题建议。

  * 若话题跨领域，发起“卓伟调研”或“其他专家 Agent”（未来可扩展）协作。

* **卓伟（调研）**：

  * 接到协作请求后执行；返回 sources；报告仅作为“内部参考”。

* **确认机制**：

  * 当数据冲突/不确定（置信度<阈值）或操作有代价（长时/多次），诸葛生成“确认卡片”：

    * 文案：是否需要进行深度调研？预计耗时/来源数/可能成本。

## 前端体验

* **来源展示**：Manus 风格，仅显示“已浏览网页”列表；不展示报告正文。

* **系统事件**：

  * 显示“委托卓伟调研”“自主搜索”等事件气泡。

  * 对 confirm\_request，提供一键“继续/取消”。

* **偏好设置**：用户可设定“自动化程度、预算上限、时效阈值”。

## 后端策略

* **模型与 Key**：统一使用 doubao-seed-1-6-251015 + 指定 Key；支持 .env 覆盖。

* **时效提示**：在 /api/agents/research 中注入当前日期与“2025-2026 优先”约束。

* **缓存与节流**：query → 5min TTL；每会话 15s 最多一次调研；失败重试一次。

* **审计与可观测**：记录工具调用原因、来源数量、错误栈；便于回溯与质量评估。

## 安全与合规

* Key 仅读自环境变量；日志中不打印敏感信息。

* 对来源链接做安全过滤；避免注入风险。

## 验证方案

* 端到端场景用例（政策/公司动态/消费趋势），关注：

  * 是否正确判断工具调用；

  * 来源是否新鲜、数量合理；

  * 总结是否简洁；选题建议是否具体可执行。

* 建立“协作轨迹”日志与质量评分面板。
