## 功能目标
- 左侧热点列表展示时间（最新在前，支持相对时间）
- 抓取结果在服务端缓存，避免每次全量抓取
- 刷新为增量更新，并提示“新增X条热点”
- 支持按平台与分类筛选已有热点

## 后端改造
- 缓存与增量更新
  - 在服务端维护内存缓存：trendStore（Map<id, item>）与聚类结果缓存。
  - 每次刷新时抓取三方数据，给每条新增项打上 fetched_at（ISO 时间），按 id 合并到缓存，计算新增数量 new_count。
  - 重新对缓存中的全部热点进行聚类（保证跨平台聚合的准确性），并按时间排序（取聚类内原始素材的最大时间作为 cluster_time）。
- 时间字段
  - 新增字段：fetched_at（抓取时间，所有来源都有）。
  - 如能解析来源时间（有则取之），否则回退到 fetched_at。
- API 设计
  - GET /api/trends/aggregated：返回当前聚类列表（含 category、platforms、materials、cluster_time）。
  - POST /api/trends/refresh：触发增量抓取与重聚类，返回 { new_count, data }。
  - 说明：筛选在前端进行，避免增加复杂查询接口。
- 聚类与排序
  - 继续使用已有 LLM 聚类；失败时回退为单条热点分组。
  - 聚类结果以 cluster_time（组内素材最大时间）降序返回。

## 前端改造
- 时间展示
  - 在 TrendList 中的平台图标与分类标签旁，新增时间标签：相对时间（例如“3小时内”/“昨天”）或精确时间。
- 分类标签（已支持）
  - 保持当前分类标签展示。
- 筛选交互
  - 在列表头部添加轻量筛选条：平台（知乎/微博/36Kr）与分类（科技/社会/娱乐/财经/生活/国际）。
  - 前端仅对已有列表做过滤，不触发后端新抓取。
- 刷新与新增提示
  - 将“刷新”按钮调用 POST /api/trends/refresh，收到 new_count 后在头部显示“新增X条热点”，并闪烁高亮新条目。

## 验证方案
- 本地启动后端与前端，首次加载应展示缓存中的聚类列表。
- 连续点击刷新，观察“新增X条热点”提示与列表数量变化。
- 切换平台与分类筛选，验证列表动态过滤且无后端请求。
- 人工断网或 LLM 超时时，确保降级策略仍返回数据并正常排序。
