# 智能选题与调研助手（诸葛 & 卓伟）实现方案

## 1. 背景与目标
目前“诸葛”在生成选题时主要依赖模型内置知识，对于实时热点或特定垂直领域的新信息缺乏了解，导致生成的选题有时缺乏深度或准确性。
用户的目标是：
1.  让“诸葛”变聪明：遇到不了解的事先搜索，不胡编乱造。
2.  引入“卓伟”：一个专业的调研 Agent，负责帮助诸葛进行信息搜集。
3.  实现自动协作：诸葛在必要时自动请求卓伟帮助，最终生成更好的选题。

## 2. 核心架构设计

采用 **主从 Agent 协作模式 (Manager-Worker Pattern)**：

*   **主 Agent（诸葛 - 诸葛亮）**：
    *   **角色**：选题策划总监。
    *   **职责**：理解用户需求，拆解选题方向，判断是否需要补充信息，整合信息生成最终选题。
    *   **核心能力**：逻辑推理、创意生成、任务编排。
*   **从 Agent（卓伟 - 狗仔/调研员）**：
    *   **角色**：专业调研员。
    *   **职责**：接收调研需求，进行联网搜索，阅读并总结相关信息，输出客观的调研报告。
    *   **核心能力**：联网搜索、信息筛选、内容摘要。

## 3. 详细实现方案

### 3.1 卓伟（Zhuowei）- 调研 Agent 实现

由于需要联网能力，我们将利用火山引擎的 **联网版模型 (Browsing Model)** 或集成搜索工具。

*   **模型选型**：`doubao-pro-32k-browsing` (利用其原生联网能力) 或 通用模型 + Search API。
    *   *推荐方案*：优先使用 `doubao-pro-32k-browsing`，开发成本低，效果直接。
*   **System Prompt (人设定义)**：
    ```markdown
    你叫卓伟，是一个专业的全网信息调研员。你的任务是根据用户的调研需求，通过网络搜索获取最新、最准确的信息，并整理成一份简明扼要的调研报告。
    
    要求：
    1. **客观真实**：只陈述搜索到的事实，不进行主观臆测。
    2. **来源标注**：尽可能标注信息来源。
    3. **结构清晰**：使用列表或分段陈述核心发现。
    4. **针对性强**：只回答调研需求相关的内容，不要废话。
    ```
*   **API 接口设计**：
    *   Endpoint: `/api/agents/research`
    *   Input: `{ "query": "调研主题或问题" }`
    *   Output: `{ "report": "调研报告内容...", "sources": [...] }`

### 3.2 诸葛（Zhuge）- 选题 Agent 升级

我们需要赋予诸葛“意识到自己知识盲区”并“发起调研请求”的能力。

*   **ReAct (Reasoning + Acting) 模式改造**：
    诸葛的思考过程将分为两步：
    1.  **Thinking**: 分析用户需求，判断是否需要外部信息。
    2.  **Action**: 
        *   如果通过，直接生成选题。
        *   如果缺信息，生成特定指令调用卓伟。

*   **System Prompt 升级**：
    ```markdown
    ...（原有诸葛人设）...
    
    【能力升级】
    你现在拥有一个名叫“卓伟”的专业调研助手。当你发现用户提到的概念、事件或领域你不够了解，或者需要最新的数据支持时，**不要编造信息**，请立即向卓伟下达调研指令。
    
    【交互格式】
    如果你需要调研，请只输出以下 JSON 格式的指令（不要输出其他内容）：
    {
      "tool_call": "research",
      "query": "需要卓伟去调研的具体问题"
    }
    
    如果信息充足，请直接按照原有格式输出选题方案。
    ```

### 3.3 交互流程 (Orchestration)

前端或后端中间件需要处理诸葛的返回结果：

1.  **用户**：输入“帮我策划几个关于 DeepSeek 的选题”。
2.  **诸葛**：(内部思考：DeepSeek 是最近的热点，我需要更多细节) -> 输出 `{"tool_call": "research", "query": "DeepSeek 最近的大新闻和核心特点"}`。
3.  **系统 (Controller)**：
    *   检测到 `tool_call`。
    *   在 UI 上显示状态：“诸葛正在委托卓伟进行调研...”
    *   调用 **卓伟 API** (`/api/agents/research`)。
4.  **卓伟**：(联网搜索) -> 返回“DeepSeek 是幻方量化旗下的 AI 模型，最近发布了 V3 版本，特点是开源、低成本...”
5.  **系统**：将卓伟的报告作为“补充上下文”再次发送给诸葛。
    *   Prompt: `用户需求：...。卓伟的调研报告：[DeepSeek...信息]。请基于以上信息生成选题。`
6.  **诸葛**：基于报告生成高质量选题。

## 4. 开发计划 (Action Plan)

### Phase 1: 基础设施建设
1.  **后端**：在 `server/index.cjs` 中新增 `/api/agents/research` 接口，对接火山引擎联网模型。
2.  **后端**：优化 `/api/agents/test` 接口，确保其支持更长的 Context Window（用于容纳调研报告）。

### Phase 2: 卓伟 Agent 实现
1.  **Prompt 调试**：编写并测试卓伟的 System Prompt，确保其输出格式稳定。
2.  **联网测试**：验证 `doubao-pro-32k-browsing` 的搜索效果。

### Phase 3: 诸葛 Agent 升级与前端适配
1.  **前端 (ChatInterface)**：
    *   增加对 `tool_call` JSON 格式的解析逻辑。
    *   增加“调研中”的 UI 状态（如显示两个头像协作的动画）。
    *   实现多轮对话逻辑（User -> Zhuge -> Tool -> Zhuge -> Final Reply）。
2.  **Prompt 迭代**：调整诸葛的 Prompt，使其能够准确判断何时调用卓伟。

### Phase 4: 验证与优化
1.  **集成测试**：测试典型场景（如“最近很火的 Sora 怎么看”），验证是否触发调研。
2.  **效果评估**：对比开启与关闭卓伟调研时的选题质量。

## 5. 技术栈选型
*   **LLM Provider**: Volcengine (火山引擎)
    *   诸葛: `doubao-pro-32k` (逻辑强)
    *   卓伟: `doubao-pro-32k-browsing` (联网能力)
*   **Backend**: Node.js + Express (现有架构)
*   **Frontend**: React (现有架构)
