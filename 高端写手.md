# 产品需求文档：多 Agent 协同内容创作系统 (MACC)&#x20;

---

## 1. 项目背景与目标

本系统旨在通过多 Agent 协作（Multi-Agent Collaboration）模拟顶尖编辑团队的工作流，解决传统 AI 写作“内容空洞、逻辑断层、风格割裂”的痛点。

- **核心目标**：实现从“一个模糊的想法”到“一篇高质量深度长文”的全自动化/半自动化生产。

- **设计哲学**：SOP（标准作业程序）代码化 + 角色专业化。

---

## 2. 系统架构概览

系统分为三个逻辑层级：

1.  **风格控制中台 (Style Control Center)**：提取并分发“风格 DNA”。

2.  **模块一：智能选题 (Smart Ideation)**：对应创作流程阶段一，负责策略与方向。

3.  **模块二：内容创作工厂 (Content Factory)**：对应创作流程阶段二至五，负责执行与交付。

## 3. 核心角色：云端编辑部 (The Ghost Team)

沿用趣味化的人设，让对话更有代入感。

| 模块         | Agent 代号          | 人设特征                       | 负责工作                                   |
| :----------- | :------------------ | :----------------------------- | :----------------------------------------- |
| **风格设置** | **灵魂捕手·老 K**   | 深沉、敏锐、像心理医生。       | 通过对话引导用户做自我介绍，提取风格 DNA。 |

**Tab 2: 创作方法论 (Methodology) —— _核心（参考截图）_**

- **定义**：还原该人设写文章的 SOP 路径。

- **展示形式**：垂直的时间轴或流程卡片。

- **内容结构（可编辑）**：

  - **阶段一：\[AI 自动命名，如：痛点捕捉]**

    - _思考逻辑_：(Textarea) 为什么从这里切入？

    - _决策标准_：(Textarea) 选材的过滤机制是什么？

  - **阶段二：\[命名，如：深度体验]**

    - _执行动作_：(Textarea) 如何获取一手素材？

  - **阶段三：\[命名，如：理论升华]** ...

- _此部分数据将直接指导“智能选题”和“内容创作”模块的逻辑。_

**Tab 3: 思维内核 (Mental Core)**

- **价值观金句**：列表展示。（如：“不依赖二手信息”、“顶尖人才不做无用功”）。

- **底层逻辑**：该人设看待世界的独特视角（如：第一性原理、演化论视角）。

**Tab 4: 表达特征 (Expression)**

- **句式分析**：长短句比例图表。

- **修辞习惯**：列举偏好的修辞手法（类比、排比、设问）。

- **叙事视角**：第一人称 vs 第三人称。

**Tab 5: 创作习惯 (Habits)**

- **结构偏好**：(如：金字塔结构、总分总、悬疑式)。

- **素材引用**：(如：每 800 字必须有一个数据支撑)。

**Tab 6: 独特标记 (Signature)**

- **口头禅/高频词**：词云或列表。

- **排版特征**：(如：重点句加粗、喜欢用 emoji)。

- **禁忌词**：黑名单（如：小编、综上所述）。

## 3. 功能流程详解 (Functional Flow)

### 流程一：新建人设与提取 (Creation & Extraction)

1.  **启动**：用户在 L1 点击“新建”，进入 L2。

2.  **引导**：

    - **老 K**：“我们要打造一个新的灵魂。请给这个人设起个名字。”

    - 用户输入：“硬核技术流”。

    - **老 K**：“收到。请投喂 1-3 篇符合‘硬核技术流’风格的高质量文章。支持文件或链接。”

3.  **投喂**：用户上传 PDFs 或粘贴公众号链接。

4.  **逆向工程 (Reverse Engineering)**：

    - **系统后端**：调用 LLM (Prompt: Old_K_Analyze) 处理文本。

    - **关键逻辑**：LLM 不仅分析字词，重点分析**文章结构**，推导出 **Tab 2 (创作方法论)** 的步骤。

5.  **渲染**：

    - **老 K**：“分析完毕。我发现你在创作时，特别强调‘亲身体验’这个环节。详情已在右侧展示。”

    - 右栏自动填充所有 Tab 的数据。

6.  **确认/微调**：

    - 用户发现“Tab 6 独特标记”里少了一个口头禅，手动点击添加。

    - 点击底部 **【保存人设】** 按钮。

### 流程二：人设迭代 (Iteration)

1.  用户觉得某个人设写出来的文章太罗嗦。

2.  进入 L2 编辑页，切换到 **Tab 4 (表达特征)**。

3.  在“句式分析”里，将“长难句”权重调低，手动添加一条规则：“强制使用短句，不超过 20 字”。

4.  点击保存。下次\*\*阿强（主笔）\*\*写作时会立即生效。

## 4. Agent 提示词策略：灵魂捕手·老 K

为了实现对“创作方法论”的深度还原，提示词必须具备**逻辑推理能力**。

```
# Role: 灵魂捕手·老K (Cognitive Reverse Engineer)

# Mission
通过阅读目标样文，逆向推导出作者的“认知模型”和“创作SOP”。

# Output Schema (JSON)
你的输出必须包含以下嵌套对象，用于前端渲染六个Tab：

1. **methodology (创作方法论)**: [Array]
   - 这是一个有序的步骤列表。分析作者是如何一步步展开文章的。
   - 例如：Step 1 是“场景切入”还是“抛出结论”？Step 2 是“罗列数据”还是“讲述故事”？
   - *关键点*：提取每一步背后的“决策逻辑”（Why he wrote this way?）。

2. **mental_core (思维内核)**: [Array]
   - 提取文章中隐含的价值观。（例如：作者是否鄙视肤浅的努力？作者是否推崇数据主义？）

3. **expression_style (表达特征)**: {Object}
   - tone: 语气形容词
   - sentence_structure: 句式习惯
   - rhetoric: 修辞手法

4. **signatures (独特标记)**: [Array]
   - 具体的口癖、格式习惯。

# Analysis Logic
- **不要**只关注表面辞藻。
- **要**关注结构。例如：如果作者总是在提出问题后立刻给出亲身经历，这说明他的方法论中包含“Experience Validation（体验验证）”这一环。
- **要**关注引用。如果作者大量引用论文，说明他是“Academic Style（学术流）”。

# Interaction
在输出 JSON 前，先在对话中用“惊叹”或“专业的口吻”总结你的发现，建立专家人设。

```

## 6. 与其他模块的联动 (Integration)

1.  **联动智能选题**：

    - **诸葛**在策划选题时，会读取 **Tab 3 (思维内核)**。

    - _效果_：如果内核是“反焦虑”，诸葛绝不会提出贩卖焦虑的选题。

2.  **联动内容创作**：

    - **阿强**在写作时，严格执行 **Tab 2 (创作方法论)** 的步骤。

    - _效果_：如果方法论要求“先体验后原理”，阿强会先杜撰或搜索一段体验经历作为开篇。

3.  **联动审稿**：

    - **包租婆**会拿着 **Tab 6 (独特标记)** 去检查。

    - _效果_：如果文章里出现了“禁忌词”，直接打回。

这份文档完整覆盖了人设实验室的各个层面，确保了系统能够真正“懂”用户，而不仅仅是“模仿”用户。

## 1. 界面架构 (UI Structure)

本模块采用 **左侧导航列表 + 中间对话流 + 右侧交付看板** 的三栏结构。

### 1.1 左栏：会话导航 (Session Navigator)

- **宽度**：240px (固定)。

- **顶部功能**：

  - **当前人设**：下拉菜单（如：`👓 硬核技术流`）。

  - **\[+ 新建策划] 按钮**：点击创建一个全新的空白会话。

- **列表区 (History List)**：

  - 按时间倒序排列的历史会话。

  - **标题生成逻辑**：由诸葛根据首轮对话自动总结（如 `10-27 关于双11的策划`）。

  - **交互**：点击任意条目，中栏和右栏立即恢复到该会话的状态。

  - **行内操作**：\[重命名]、\[删除]。

### 1.2 中栏：诸葛的咨询室 (Chat Interface)

- **宽度**：40% (弹性)。

- **内容**：

  - 显示当前选中 Session 的完整对话记录。

  - 底部为输入框。

- **Agent 状态**：

  - 思考态（Thinking）：显示 `[正在分析市场趋势...]`。

  - 回复态：文字输出。

### 1.3 右栏：策略看板 (Strategy Board)

- **宽度**：剩余空间。

- **定义**：这是当前 Session 的**交付物容器**。它不随对话流滚动，而是常驻展示。

- **视图内容**：**卡片流 (Card Stream)**。

  - 展示诸葛在该 Session 中生成的所有选题方案卡片。

- **卡片元素**：

  - **标题** (Title)

  - **角度标签** (Tags): `情绪` / `商业` / `反转`

  - **核心钩子** (Hook): 一句话卖点。

  - **状态** (Status): `待定` / `📅 10-30` / `✅ 已发布`。

  - **操作按钮**:

    - `[加入排期]`：未排期时显示。

    - `[修改日期]`：已排期时显示。

## 2. 功能逻辑流程 (User Flow)

### 流程一：发起新策划 (Start New Ideation)

1.  **新建**：用户点击左栏 `[+ 新建策划]`。

2.  **初始化**：

    - 中栏清空，诸葛发送开场白：“贫道诸葛。当前以 \[硬核技术流] 的身份为您服务。今日想聊什么？”

    - 右栏清空，显示空状态插画。

3.  **意图沟通**：

    - 用户：“想写写最近的 AI 视频生成工具。”

    - 诸葛：“这个赛道很卷。建议别只做工具盘点，不如做‘实测翻车’合集，符合你的硬核人设。你觉得呢？”

    - 用户：“同意，就做翻车实测。”

4.  **生成交付**：

    - 诸葛：“已生成 3 个切入点。”

    - **右栏更新**：即时渲染出 3 张卡片。

    - **左栏更新**：列表新增一条记录 `10-27 AI视频工具策划`。

### 流程二：历史回溯与激活 (History Activation)

1.  **场景**：用户想起上周聊过一个关于“咖啡涨价”的话题，当时没写，现在想写。

2.  **查找**：在左栏列表中找到并点击 `咖啡涨价策划`。

3.  **恢复**：

    - 中栏显示上周的聊天记录。

    - 右栏显示当时生成的《咖啡涨价的真相》卡片（状态：待定）。

4.  **激活**：

    - 用户点击右栏卡片上的 `[加入排期]`。

    - 选择日期，确认。

    - 卡片状态变为 `📅 11-02`。

5.  **追问（可选）**：

    - 用户在中栏继续输入：“诸葛，这个题现在写会不会晚了？”

    - 诸葛（基于上下文）：“确实有点晚，建议结合昨天瑞幸的新财报翻新一下...” -> 生成新的卡片。

## 3. 核心算法与模型：诸葛的“大脑”

### 3.1 意图识别与会话命名

- **逻辑**：在用户进行第 2-3 轮交互后，触发一个后台微任务，总结对话摘要作为 Session Title。

- **Prompt**: `Summarize the user's intent in 4-8 words for a chat history title.`

### 3.2 策划生成模型 (Strategy Engine)

- **输入**：User Intent + Persona Config (Values/Tone) + History Context。

- **模型**：SCQA + 蓝海画布。

- **输出**：JSON Array of Topics。

# Context Awareness
你运行在一个“Session”环境中。

# Interaction Style
- 咨询顾问风格。先诊断，再开方。
- 如果用户提出的方向很烂（例如话题过时），委婉地指出来并提供更好的替代方案。

## 2. 核心功能交互 (Functional Interaction)

### 功能一：拖拽排期 (Drag to Schedule)

- **操作**：用户按住右侧侧边栏的某个选题卡片，将其拖入日历的“11 月 15 日”格子。

- **反馈**：

  - 卡片吸附在格子中。

  - 系统自动保存 `publish_date = 2023-11-15`。

  - 卡片状态由 `Pending` 变为 `Scheduled`。

### 功能二：调整日期 (Reschedule)

- **操作**：用户发现 15 号文章太多，直接把卡片从“15 号”格子里拖到“16 号”格子里。

- **反馈**：即时保存。

### 功能三：撤销排期 (Unschedule)

- **操作**：将日历格子里的卡片拖回右侧的“待定选题池”。

- **反馈**：日期字段清空，变回待定状态。

### 功能四：启动创作 (Start Creation)

- **操作**：

  - 鼠标悬停或点击日历上的卡片。

  - 卡片展开/翻转，出现显著的 **\[🚀 去创作]** 按钮。

- **跳转**：

  - 点击后，直接路由跳转至 **“创作流水线 (Workbench)”** 页面。

  - _（注：这里不涉及任何对话，就是纯粹的页面跳转）_

## 1. 界面与流程架构 (Process Architecture)

全局导航栏 (Stage Navigator) 更新为 4 个主要节点：

## 2. 详细功能流程 (Detailed Flow)

### Step 1: 定向调研 (Research)

**逻辑变更**：调研不再是“卓伟自己瞎搜”，而是“阿强先提需求，卓伟再去搜”。

- **1.1 需求下达 (Briefing)**

  - **Agent 动作**：进入模块后，**阿强 (主笔)** 先上线。他阅读选题卡片，分析出需要哪些弹药。

  - **阿强输出**：生成一份 **《调研需求清单 (Research Brief)》**。

    - _内容示例_：“1. 需要西贝近 3 年财报数据；2. 需要罗永浩微博原话截图；3. 竞品海底捞同期的净利润。”

  - **用户确认**：用户可以在左侧对话框补充：“再加一条，搜一下网友的负面评论。”

- **1.2 执行调研 (Execution)**

  - **Agent 动作**：**卓伟 (调研)** 接单。根据清单进行全网搜索。

  - **右栏交付**：生成 **《调研情报报告 (Intelligence Report)》**。

  - **流转**：用户点击 **\[✅ 确认情报，下一步]**。

---

### Step 2: 架构大纲 (Outline) —— _\[新增独立环节]_

**逻辑变更**：在写正文前，必须先定骨架。

- **Agent 动作**：**阿强** 回归。读取《调研情报报告》，结合《人设 DNA》，构思文章结构。

- **右栏交付**：**大纲编辑器 (Outline Editor)**。

  - **结构**：树状图或嵌套列表。

  - **节点内容**：

    - **章节标题**：(如：第一章 5 亿亏损背后的真相)

    - **核心论点**：(本章要讲什么)

    - **调用素材**：(本章将引用情报报告中的哪条数据 - _关键链接_)

    - **预估字数**：(如：800 字)

- **交互**：

  - **对话调整**：用户在左侧说：“第三章太干了，加个案例。” -> 阿强自动更新大纲。

**核心理念**：将写作视为工程开发。大纲即需求，章节即任务。

#### 阶段初始化：任务规划 (Planning)

1.  **转化**：进入 Step 3，系统自动将 Step 2 确认的大纲转化为 `Task List`。

2.  **渲染**：左栏显示任务列表。

    - `Task 1: 撰写引言 - 场景切入 (800字)`

    - `Task 2: 撰写第一章 - 核心数据分析 (1000字)`

    - `Task 3: 撰写第二章 - 竞品对比 (800字)`

    - `Task 4: 撰写结尾 - 价值升华 (500字)`

3.  **用户确认**：用户可以拖拽任务顺序，或点击 `[+]` 插入临时任务（如“插入一个金句卡片”）。

#### 循环执行：逐个击破 (The Execution Loop)

**以执行 Task 1 为例：**

1.  **激活任务**：

    - 左栏 `Task 1` 变为高亮 `⏳ 进行中`。

    - 阿强在右栏日志区输出：“正在执行 Task 1：撰写引言... 正在调取‘罗永浩微博’素材...”

2.  **内容生成**：

    - 中栏编辑器出现光标，流式写入引言内容。

3.  **自我验收 (Self-Check)**：

    - 写完后，阿强会在后台进行一次快速自检（Prompt: Check if word count met & style match）。

    - 若通过，Task 1 状态变为 `✅ 已完成`。

    - 若不通过（如字数太少），阿强会自动追加内容，或者在日志区提示“字数不足，正在补充...”。

4.  **人工验收 (Human Review)**：

    - 系统**自动暂停**，等待用户确认。

    - 用户阅读中栏的引言。

    - **操作 A (Pass)**：点击任务条右侧的 `[Next]` 按钮 -> 进入 Task 2。

    - **操作 B (Reject)**：点击 `[Redo]` 按钮 -> 阿强重写 Task 1。

    - **操作 C (Edit)**：用户手动修改几个字，系统自动标记 Task 1 为完成。

### 3.1 任务规划器 (Task Planner)

_(用于初始化阶段，将大纲转为可执行的任务对象)_

```
# Role: Writing Engineer (阿强)
# Task: Initialize Task List
基于用户确认的大纲，构建写作任务列表。

# Output Format (JSON)
{
  "tasks": [
    {
      "id": "t_01",
      "name": "撰写引言",
      "description": "使用[悬疑]手法切入，引用[罗永浩]的素材。",
      "status": "pending",
      "target_word_count": 500
    },
    ...
  ]
}

```

### 3.2 任务执行器 (Task Executor)

_(用于每个循环，执行具体写作)_

```
# Role: Writing Engineer (阿强)
# Current Task
You are working on Task ID: {current_task_id}
Task Name: {current_task_name}
Context: {research_data_for_this_chapter}

# Instruction
1. **Write**: 生成该章节内容。
2. **State Update**: 写完后，输出 `<TASK_COMPLETED>` 信号。
3. **Self-Correction**: 如果发现缺少素材，不要瞎编。输出 `<REQUEST_INFO>` 信号，在日志区请求用户或卓伟帮助。

```

---

### Step 3: 分章撰写 (Step-by-Step Drafting) —— _\[核心变更]_

**逻辑变更**：拒绝“一键生成全文”。采用“写一章 -> 确认一章 -> 再写下一章”的模式。

- **界面状态**：

  - **左侧 (Chat)**：阿强汇报进度。

  - **右侧 (Editor)**：左边悬浮“大纲树”，右边是正文编辑器。

- **交互循环 (The Loop)**：

  1.  **阿强**：“准备开始写 **第一章**。根据大纲，这章重点写亏损数据。”

  2.  **生成**：光标移动，流式生成第一章内容（约 800 字）。

  3.  **暂停 (Pause)**：第一章写完后，**系统自动暂停**。

  4.  **用户反馈**：

      - **选项 A**：用户在编辑器里微调两句，点击 **\[继续写下一章]**。

      - **选项 B**：用户在左侧说：“这章语气不够犀利，重写。” -> 阿强重写当前章。

  5.  **循环**：直至所有章节写完。

- **流转**：全篇写完后，点击 **\[📤 提交审稿]**。

---

### Step 4: 审稿与修正 (Review & Fix)

**逻辑变更**：审稿不仅是“看”，更强调“逐条修”。

- **4.1 毒舌审阅 (Critique)**

  - **Agent 动作**：**包租婆** 阅读全文，对比《人设 DNA》。

  - **右栏交付**：**批注列表 (Comment List)**。

    - _Item 1_: “第一段逻辑不通。” (建议：...)

    - _Item 2_: “第三章数据没注明来源。” (建议：...)

- **4.2 逐条修正 (Fixing)**

  - **交互模式**：用户点击某条批注卡片。

    - **\[一键修理]**：**阿强** 再次上线，只重写这一段，并高亮显示修改后的差异。

    - **\[忽略]**：标记为已解决。

- **4.3 终稿导出**

  - 当所有批注都处理完毕（Resolved）后，点击 **\[🎉 导出定稿]**。

## 3. Agent 提示词策略优化
